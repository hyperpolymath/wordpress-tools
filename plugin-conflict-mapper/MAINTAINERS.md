## Becoming a Maintainer

Maintainers are added through the following process:

### Criteria

- **Sustained Contribution**: Regular, high-quality contributions over 6+ months
- **Community Engagement**: Helpful in issues, PRs, and discussions
- **Technical Expertise**: Demonstrated deep understanding of the codebase
- **Reliability**: Consistent, timely responses to project needs
- **Values Alignment**: Commitment to Code of Conduct and project values

### Process

1. **Nomination**: Current maintainer nominates candidate
2. **Discussion**: Maintainers discuss candidate's qualifications
3. **Approval**: Unanimous approval from existing maintainers
4. **Invitation**: Lead maintainer extends invitation
5. **Onboarding**: New maintainer gets access and onboarding

### New Maintainer Onboarding

- Repository commit access
- Access to communication channels
- Introduction to maintainer workflows
- Mentorship from existing maintainer

## Maintainer Expectations

### Time Commitment

- **Minimum**: 2-4 hours per week
- **Reviews**: Respond to PRs within 72 hours
- **Issues**: Triage new issues within 48 hours
- **Security**: Respond to security issues within 24 hours

### Communication

- Check project channels at least weekly
- Communicate availability and time off
- Participate in maintainer discussions
- Be respectful and professional

### Decision Making

- **Consensus**: Prefer consensus on major decisions
- **Disagreement**: Respectful discussion, then vote if needed
- **Veto**: Lead maintainer has final veto on security/legal issues
- **Documentation**: Document major decisions in issues/discussions

## Inactive Maintainers

Maintainers inactive for 6+ months may be:

1. Contacted to check status
2. Moved to inactive list
3. Have permissions reduced

They can be reinstated upon return.

## Removing Maintainers

In rare cases, maintainers may be removed for:

- **Code of Conduct violations**
- **Security breaches**
- **Sustained inactivity** (12+ months without communication)
- **Loss of trust** (requires unanimous vote)

Process:
1. Private discussion among other maintainers
2. Attempt to resolve issues
3. If unresolved, unanimous vote to remove
4. Private notification with reasoning

## Contributor Ladder

We recognize contribution at multiple levels:

### 1. Contributor
- Anyone who submits a pull request
- Listed in GitHub contributors

### 2. Regular Contributor
- 3+ merged PRs
- Acknowledged in CHANGELOG

### 3. Trusted Contributor
- 10+ merged PRs
- Can be assigned issues
- Input valued in discussions

### 4. Committer
- 25+ merged PRs
- Commit access to feature branches
- Can approve PRs

### 5. Maintainer
- Meets maintainer criteria
- Full commit access

## Project Governance

### Governance Model

This project uses a **Benevolent Dictator** model:

- **Lead Maintainer** makes final decisions
- **Consensus preferred** among all maintainers
- **Community input** valued and considered
- **Transparency** in decision-making process

### Decision Authority

| Decision Type | Authority | Process |
|--------------|-----------|---------|
| Bug fixes | Any maintainer | Review + merge |
| New features | Maintainer consensus | Discussion + vote |
| Breaking changes | Lead + consensus | RFC + community input |
| Security issues | Lead maintainer | Immediate action |
| Maintainer addition | Unanimous vote | Nomination + approval |
| Maintainer removal | Unanimous vote | Discussion + vote |

### Conflict Resolution

1. **Discussion**: Talk through disagreement
2. **Mediation**: Another maintainer mediates
3. **Vote**: If unresolved, vote (simple majority)
4. **Final Decision**: Lead maintainer if vote ties

## Tri-Perimeter Contribution Framework (TPCF)

This project operates under TPCF governance:

### Perimeter 1: Core (Trusted)
- **Members**: Core maintainers
- **Access**: Full repository access
- **Trust**: High - extensive vetting
- **Permissions**: Merge, release, admin

### Perimeter 2: Contributor (Verified)
- **Members**: Regular contributors
- **Access**: Fork and PR workflow
- **Trust**: Medium - proven track record
- **Permissions**: Create PRs, review code

### Perimeter 3: Community (Sandbox)
- **Members**: Anyone
- **Access**: Public issues and discussions
- **Trust**: Low - unvetted
- **Permissions**: Report issues, participate in discussions

Contributors move between perimeters based on demonstrated trust and contribution history.
